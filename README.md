# Alpha Touring Challenge ‚Äî Website

Stack: [Astro](https://astro.build) (hybrid static/on-demand rendering) + [Tailwind CSS](https://tailwindcss.com) + [Supabase](https://supabase.com) (Postgres, queried via plain `fetch` against its REST API ‚Äî no SDK dependency) + [Cloudflare Pages](https://pages.cloudflare.com) hosting.

Roster and news pages are rendered on-demand at Cloudflare's edge (`export const prerender = false`) with a 60-second cache, so editing data in Supabase shows up on the site within about a minute ‚Äî no rebuild or redeploy required. The homepage's news teaser works the same way. Everything else is static.

## First-time setup

1. **Install dependencies**
   ```
   npm install
   ```

2. **Create a Supabase project** at supabase.com (free tier is enough for this). In the SQL editor, run, in order:
   - `supabase/migrations/0001_init.sql` ‚Äî creates all tables, lookups, and RLS policies
   - `supabase/seed/seed_teams.sql`
   - `supabase/seed/seed_drivers.sql`
   - `supabase/seed/seed_news.sql`

   (`seed_teams.sql` and `seed_drivers.sql` are generated from the roster spreadsheet ‚Äî see below.)

3. **Copy `.env.example` to `.env`** and fill in your Supabase project's URL and anon key (Supabase dashboard ‚Üí Settings ‚Üí API), plus the Discord/Redbubble links (already filled in with the real ones).

4. **Run locally**
   ```
   npm run dev
   ```

5. **Deploy**: connect this repo in the Cloudflare dashboard under Workers & Pages ‚Üí Create application ‚Üí Pages tab ‚Üí Import an existing Git repository (build command `npm run build`, output directory `dist`). Add the same `.env` variables under the project's Settings ‚Üí Environment variables. Every `git push` to your main branch redeploys automatically.

   Cloudflare's Git-connected builds now deploy via `wrangler deploy` rather than the older Pages-specific bundler, so `wrangler.jsonc` at the repo root (already included) is required ‚Äî it tells Wrangler where the built worker (`dist/_worker.js/index.js`) and static assets (`dist/`) are. Bump `compatibility_date` in that file occasionally (any date is fine as long as it's in the past). Two things baked into this repo that fix errors Wrangler otherwise throws on deploy:
   - `public/.assetsignore` (copied into `dist/` on every build) tells Wrangler not to upload the `_worker.js` server bundle as a public static asset ‚Äî it's still used as the Worker's entry point via `main`, just excluded from the public asset manifest.
   - `wrangler.jsonc`'s `vars` block pins the three public URL values so `wrangler deploy` doesn't strip them from the Worker's config on every deploy. `PUBLIC_SUPABASE_ANON_KEY` is deliberately left out of this file ‚Äî keep that one set via the Cloudflare dashboard's Environment Variables only, since that's what actually feeds the `npm run build` step.
   - If Wrangler ever warns about a Worker name mismatch, update `wrangler.jsonc`'s `"name"` field to match whatever your Cloudflare dashboard project is actually named.

## Re-importing the roster later

Whenever the roster spreadsheet changes:
```
python3 supabase/seed/generate_seed.py path/to/roster.xlsx
```
This regenerates `supabase/seed/seed_teams.sql` and `seed_drivers.sql`. Re-run them in the Supabase SQL editor ‚Äî both are upserts, safe to run repeatedly.

## A note on this project's first build

This was scaffolded in a sandboxed session without access to the npm registry, PyPI, or Supabase's API (only GitHub was reachable), so none of it could be `npm install`ed or run in that session ‚Äî every file was hand-written instead of generated by `npm create astro` and friends. It's worth running `npm install && npm run build` yourself as a first check before deploying, in case anything needs a small fix.
